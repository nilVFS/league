# HotV React Site

Сайт и служебка для приватной лиги Path of Exile 2.

Если совсем по-простому: это не просто лендинг, а набор страниц + админка + Twitch-бот + автоматический импорт клипов.

Коммиты я писал ногами, мне пофиг. Если история выглядит криво, это часть лора.

## Что вообще умеет проект

Проект состоит из нескольких частей.

### 1. Публичный сайт

На сайте есть страницы:

- главная;
- ладдер;
- клипы;
- участники;
- награды.

Туда можно выводить контент из админки, без ручного редактирования кода на каждую мелочь.

### 2. Админка

Админка нужна, чтобы управлять данными сайта.

Через неё можно:

- добавлять и редактировать клипы;
- добавлять и редактировать участников;
- добавлять и редактировать награды;
- вести белый список игроков ладдера;
- смотреть заявки и спорные случаи;
- смотреть состояние Twitch-бота и подключённых каналов.

### 3. Twitch-бот

В проекте есть интеграция с Twitch.

Бот умеет:

- подключаться к Twitch через OAuth;
- подписываться на чат конкретных стримеров через Twitch EventSub;
- слушать сообщения в чатах подключённых каналов;
- обрабатывать команды для ладдера;
- отвечать в чат сообщением-подтверждением.

Важно: бот не "сидит" отдельным процессом как классический IRC-бот. Здесь логика построена через Twitch EventSub webhook.

То есть схема такая:

1. Twitch шлёт webhook в наш API.
2. API проверяет подпись.
3. Если в сообщении команда для ладдера, она разбирается и сохраняется.
4. Бот отправляет ответ в чат через Twitch API.

### 4. Команды для ладдера из Twitch-чата

Сейчас бот понимает команду такого вида:

```text
!выполнил Player#1234 15
!выполнил Player#1234 15 https://proof-link
```

Также поддерживается короткая форма:

```text
!в Player#1234 15
```

Что делает эта команда:

- пытается найти игрока по тегу;
- сверяет его с белым списком ладдера;
- ищет награду по номеру;
- создаёт или обновляет запись о выполненном достижении;
- если игрок не найден в белом списке, отправляет кейс на модерацию.

### 5. Автоматический импорт клипов

В проекте есть крон-задача, которая сама ходит в Twitch и подтягивает свежие клипы участников.

Как это работает:

1. Из коллекции `participants` берутся ссылки на Twitch-каналы участников.
2. Для каждого канала запрашиваются свежие клипы через Twitch API.
3. Берутся клипы только по нужной игре, по умолчанию `Path of Exile 2`.
4. Новые клипы сохраняются в коллекцию `clips`.
5. Уже импортированные клипы повторно не добавляются.

Сейчас окно поиска клипов - последние 12 часов.

### 6. Где живёт cron

Важно: рабочая автоматизация завязана не только на Vercel.

Судя по коду, у нас есть сценарий для таймера в Яндекс Облаке:

- сервер умеет принимать `POST /` с payload от `yandex.cloud.events.serverless.triggers.TimerMessage`;
- если внутри payload лежит `import-clips`, сервер запускает импорт клипов;
- для этого в `server/index.js` уже есть отдельная логика.

То есть по-человечески:

- крон у нас может приходить из Яндекса;
- он будит сервер;
- сервер запускает `api/cron/import-clips`.

В репозитории при этом ещё лежит `vercel.json` с cron на `api/cron/import-clips`, так что Vercel-конфиг тоже есть. Но если смотреть на код и ваш контекст, реальный сценарий "таймер -> импорт" у вас завязан на Яндекс.

## Как устроен проект

Если открываешь репу впервые, вот главное:

- `src/` - фронтенд на React + Vite;
- `api/` - серверные API-роуты;
- `server/index.js` - локальный Node-сервер, который умеет поднимать API без Vercel;
- `.env.example` - пример переменных окружения;
- `Dockerfile.api` - контейнер только для API.

Самые важные API-роуты:

- `/api/content` - чтение и запись контента;
- `/api/admin/login`, `/api/admin/logout`, `/api/admin/session` - вход в админку;
- `/api/twitch/auth/start` - старт OAuth для бота или стримера;
- `/api/twitch/auth/callback` - callback после Twitch OAuth;
- `/api/twitch/eventsub/chat` - webhook для Twitch EventSub;
- `/api/twitch/eventsub/register` - ручная регистрация подписки на чат;
- `/api/cron/import-clips` - импорт клипов;
- `/api/ladder/submit` - служебная отправка данных ладдера.

## Где хранятся данные

У проекта два режима хранения данных.

### Вариант 1. Локальный JSON-файл

Если YDB не настроена, проект хранит данные в файле.

По умолчанию это:

```text
.data/content-store.json
```

Это удобно для:

- локальной разработки;
- тестового запуска;
- копии проекта "поднять у себя и посмотреть".

### Вариант 2. YDB

Если заданы переменные `YDB_*`, проект использует YDB.

Это уже нормальный вариант для продового хранения.

## Как поднять проект у себя, совсем по шагам

### Что нужно заранее

Нужно, чтобы у тебя были:

- Node.js;
- npm;
- доступ к переменным окружения, если хочешь включать Twitch, админку и YDB.

### Шаг 1. Установить зависимости

```bash
npm install
```

### Шаг 2. Создать `.env`

```bash
cp .env.example .env
```

### Шаг 3. Если хочешь просто посмотреть фронт

Можно вообще начать с этого:

```bash
npm run dev
```

Обычно фронт будет доступен на:

```text
http://localhost:5173
```

Если `VITE_API_BASE_URL` пустой, фронт будет стучаться в API на том же домене. Для чисто локальной разработки чаще удобнее поднять API отдельно и явно прописать его адрес.

### Шаг 4. Поднять локальный API

```bash
npm run start:api
```

По умолчанию API поднимется на:

```text
http://localhost:3000
```

Проверка:

```bash
curl http://localhost:3000/healthz
```

Должно вернуть JSON с `ok: true`.

### Шаг 5. Связать фронт и API

Для локальной разработки в `.env` обычно нужно так:

```env
VITE_API_BASE_URL=http://localhost:3000
ALLOWED_ORIGINS=http://localhost:5173
```

После изменения `.env` перезапусти фронт.

## Минимальная конфигурация, чтобы просто завелось

Если хочешь просто локально поднять проект без Twitch и без YDB, вот минимально полезный `.env`:

```env
VITE_API_BASE_URL=http://localhost:3000
ALLOWED_ORIGINS=http://localhost:5173
COOKIE_SAME_SITE=lax
COOKIE_SECURE=false
CONTENT_STORE_PATH=.data/content-store.json
ADMIN_EMAIL=admin@example.com
ADMIN_PASSWORD=change-me
ADMIN_SESSION_SECRET=put-a-long-random-string-here
```

Этого хватит, чтобы:

- открыть фронт;
- поднять API;
- войти в админку;
- хранить контент локально в JSON.

## Переменные окружения и зачем они нужны

### Для фронта

`VITE_API_BASE_URL`

- адрес API, если он живёт отдельно от фронта;
- если фронт и API на одном домене, можно оставить пустым.

### Для CORS и cookie

`ALLOWED_ORIGINS`

- список разрешённых origin через запятую;
- нужен, чтобы фронт мог ходить в API с cookie.

`COOKIE_SAME_SITE`

- режим cookie для админской сессии.

`COOKIE_SECURE`

- в `true`, если работаешь только по HTTPS;
- локально обычно `false`.

### Для админки

`ADMIN_EMAIL`

- логин админа.

`ADMIN_PASSWORD`

- пароль админа.

`ADMIN_SESSION_SECRET`

- секрет для подписи админской сессии;
- должен быть длинным и случайным.

### Для локального файлового хранилища

`CONTENT_STORE_PATH`

- путь к JSON-файлу, где хранится контент, если YDB не включена.

### Для YDB

`YDB_ENDPOINT`

- endpoint базы.

`YDB_DATABASE`

- имя или путь базы.

`YDB_SERVICE_ACCOUNT_KEY_JSON`

- JSON-ключ сервисного аккаунта целиком одной строкой.

### Для Twitch

`TWITCH_CLIENT_ID`

- client id Twitch-приложения.

`TWITCH_CLIENT_SECRET`

- client secret Twitch-приложения.

`TWITCH_BOT_USER_ID`

- user id Twitch-аккаунта бота;
- нужен, чтобы подписки и отправка сообщений шли от правильного аккаунта.

`TWITCH_EVENTSUB_SECRET`

- секрет для проверки подписи webhook от Twitch.

`TWITCH_OAUTH_STATE_SECRET`

- секрет для безопасного OAuth state.

`TWITCH_CLIP_GAME_NAME`

- имя игры для фильтрации клипов;
- по умолчанию `Path of Exile 2`.

`TWITCH_CLIP_GAME_ID`

- можно задать id игры руками, чтобы не искать его по имени.

`CRON_SECRET`

- секрет для служебных вызовов cron-роутов.

### Необязательное, но полезное

`FRONTEND_BASE_URL`

- помогает корректно редиректить обратно на фронт после Twitch OAuth;
- если не задан, проект пытается взять адрес из `ALLOWED_ORIGINS`.

## Как включить Twitch-бота

Ниже сценарий "хочу, чтобы бот реально работал".

### 1. Заполнить Twitch-переменные

Нужны как минимум:

```env
TWITCH_CLIENT_ID=
TWITCH_CLIENT_SECRET=
TWITCH_BOT_USER_ID=
TWITCH_EVENTSUB_SECRET=
TWITCH_OAUTH_STATE_SECRET=
```

### 2. Поднять API на адресе, доступном Twitch

Для EventSub webhook Twitch должен достучаться до твоего API извне.

То есть `localhost` для боевого подключения каналов не подойдёт. Нужен публичный URL.

### 3. Подключить бот-аккаунт

В проекте есть страница:

```text
/bot
```

На ней есть служебная кнопка переавторизации бота.

Она запускает OAuth с `kind=bot` и просит у Twitch такие scope:

- `user:read:chat`
- `user:write:chat`
- `user:bot`

### 4. Подключить канал стримера

На той же странице `/bot` вводится логин стримера.

Дальше происходит такой флоу:

1. Стример проходит Twitch OAuth.
2. Приложение получает подтверждение.
3. Создаётся или переиспользуется подписка `channel.chat.message`.
4. Канал сохраняется в коллекцию `trackedChannels`.

Для стримера запрашивается scope:

- `channel:bot`

### 5. Проверить, что webhook живой

Webhook для сообщений чата:

```text
/api/twitch/eventsub/chat
```

Если Twitch начнёт слать события, проект:

- проверит подпись;
- поймёт, к какому каналу относится событие;
- разберёт сообщение;
- при необходимости сохранит достижение или заявку на модерацию;
- попытается ответить в чат.

## Как работает импорт клипов

Импорт клипов живёт в:

```text
api/cron/import-clips.js
```

Что он делает:

1. Берёт всех участников из коллекции `participants`.
2. Достаёт из них Twitch login.
3. Получает app access token Twitch.
4. Ищет Twitch user id для этих логинов.
5. Ищет клипы за последние 12 часов.
6. Отфильтровывает клипы по нужной игре.
7. Сохраняет новые клипы в `clips`.

Что важно знать:

- импорт идёт только методом `GET`;
- при наличии `CRON_SECRET` маршрут защищён;
- секрет можно передать через query `?secret=...` или через `Authorization: Bearer ...`.

Ручная проверка локально:

```bash
curl "http://localhost:3000/api/cron/import-clips?secret=YOUR_SECRET"
```

## Как связан Яндекс cron

Если вы разворачиваете API в Яндекс Облаке и хотите будить импорт клипов таймером, логика уже предусмотрена.

Сценарий такой:

1. Таймер в Яндексе шлёт событие в корень сервера.
2. Сервер получает `POST /`.
3. Видит тип `yandex.cloud.events.serverless.triggers.TimerMessage`.
4. Проверяет, что payload равен `import-clips`.
5. Внутри вызывает обработчик импорта клипов.

То есть отдельно писать эту склейку не нужно, она уже есть в `server/index.js`.

## Про Vercel

В репозитории есть `vercel.json`.

Он нужен для двух вещей:

- переписывать не-API роуты на `index.html`, чтобы работал React Router;
- дать Vercel-совместимую конфигурацию API.

Но если у вас боевой cron уже живёт в Яндексе, считай Vercel тут скорее совместимость и альтернативный вариант деплоя, а не единственный источник правды.


## С чего начать новому человеку

Если репу скопировал человек вообще без контекста, нормальный порядок такой:

1. Поставить зависимости через `npm install`.
2. Скопировать `.env.example` в `.env`.
3. Заполнить минимум для локального запуска.
4. Поднять API через `npm run start:api`.
5. Поднять фронт через `npm run dev`.
6. Проверить админку.
7. Потом уже отдельно настраивать Twitch, OAuth, EventSub и cron.

Если цель просто "посмотреть сайт и покликать админку", Twitch и YDB можно не трогать вообще.
